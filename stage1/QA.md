# Stage1 Q&A

## Q: 单个文件的最大大小是多少？

### 计算过程

当前 inode 只有**直接块指针**，没有间接块指针：

```c
struct d_inode {
    ...
    uint32_t direct[12];  // 12 个直接块指针
    ...
};
```

```
最大文件大小 = 12 × BLOCK_SIZE
            = 12 × 4096
            = 49,152 字节
            = 48 KB
```

### 瓶颈分析

| 因素 | 容量 | 是否瓶颈 |
|------|------|----------|
| 直接块指针数 | 12 个 | ✅ **是** |
| `size` 字段 (uint32_t) | 最大 4GB | ❌ 远未触及 |
| 数据区总容量 | 14332 × 4KB ≈ 56MB | ❌ 远未触及 |
| 数据位图 bits | 32768 | ❌ 远未触及 |

**唯一瓶颈是 12 个直接块指针**，文件大小被限制在 **48 KB**。

### 对比：加入间接块指针后的扩展

如果参考 ext2 的设计增加间接寻址，容量变化如下：

| 寻址方式 | 块指针数 | 可寻址大小 |
|----------|----------|------------|
| 12 直接 | 12 | 48 KB |
| + 1 级间接 | 12 + 1024 | 4 MB + 48 KB |
| + 2 级间接 | 12 + 1024 + 1024² | ~4 GB |
| + 3 级间接 | 12 + 1024 + 1024² + 1024³ | ~4 TB |

> 每个块指针 4 字节，一个 4KB 数据块可存 4096/4 = 1024 个指针，故一级间接可寻址 1024 块。

## Q: disk.img 的磁盘布局草图是什么样的？

### 总体布局（64MB 镜像）

```
磁盘块号    内容               大小
──────────────────────────────────────────
  0        引导块 (Boot)       1 块 = 4KB
  1        超级块 (Superblock) 1 块 = 4KB
  2        Inode 位图          1 块 = 4KB (32768 bits)
  3        数据块位图          1 块 = 4KB (32768 bits)
  4~2051   Inode 表            2048 块 = 8MB
 2052~    数据区              14332 块 ≈ 56MB
──────────────────────────────────────────
总计: 16384 块 = 64MB
```

### 关键计算

```
PREFIX_BLOCKS  = 4  (block 0~3)
BIT_PER_BLOCK  = 4096 × 8 = 32768
INODE_SIZE     = 256 字节

inode 表块数   = BIT_PER_BLOCK × INODE_SIZE / BLOCK_SIZE
               = 32768 × 256 / 4096
               = 2048 块

data_start_blk = PREFIX_BLOCKS + 2048 = 2052
data_blk_count = total_blk - data_start_blk
               = 16384 - 2052
               = 14332 块
```

### 各区域详解

#### Block 0 — 引导块

- 保留未用，兼容传统磁盘布局

#### Block 1 — 超级块

```c
struct superblock {
    uint32_t magic;               // 0x66757365 ("fuse")
    uint32_t block_size;          // 4096
    uint32_t total_inode;         // 32768
    uint32_t total_block;         // 16384
    uint32_t inode_table_start_blk; // 4
    uint32_t data_start_blk;      // 2052
    uint8_t  pad[4072];           // 填充至 4096 字节
};
```

#### Block 2 — Inode 位图

- 共 32768 bits，bit 号 = inode 号
- bit 0 保留（inode 0 不使用）
- bit 1 固定为 1（根目录 inode）
- 格式化后仅 bit 1（root）和 bit 2（test.txt）置 1

#### Block 3 — 数据块位图

- 共 32768 bits，bit 号 = 磁盘块号（直射映射）
- bit 0~2051 永远不被扫描（参见下方位图浪费说明）
- bit 2052 和 bit 2053 在格式化后置 1（root 数据块 + test.txt 数据块）

#### Block 4~2051 — Inode 表

```
inode 号 1 → 偏移 = 4 × 4096 + (1-1) × 256 = 16384
inode 号 2 → 偏移 = 4 × 4096 + (2-1) × 256 = 16640
...
inode 号 N → 偏移 = PREFIX_BLOCKS × BLOCK_SIZE + (N-1) × INODE_SIZE
```

每个 inode 256 字节：

```c
struct d_inode {
    uint16_t mode;          // 文件类型 + 权限 (040755 / 0100644)
    uint16_t nlink;         // 硬链接数
    uint32_t size;          // 文件大小
    uint32_t atime, mtime, ctime;
    uint32_t direct[12];    // 直接块指针
    uint8_t  pad[188];      // 填充至 256 字节
};
```

#### Block 2052~ — 数据区

- 存放文件内容和目录项
- 格式化后已分配：
  - Block 2052：根目录数据（`.` `..` `test.txt`）
  - Block 2053：test.txt 文件内容

### 格式化后的磁盘状态

```
Block 0:  [空]
Block 1:  [Superblock: magic=0x66757365, data_start=2052, ...]
Block 2:  [Inode bitmap:  bit1=1(root), bit2=1(test.txt), 其余=0]
Block 3:  [Data bitmap:   bit2052=1(root数据), bit2053=1(test.txt数据), 其余=0]
Block 4:  [Inode 1: root    — mode=040755, nlink=3, size=390, direct[0]=2052]
Block 5:  [Inode 2: test.txt— mode=0100644, nlink=1,   size=46,  direct[0]=2053]
Block 6~2051: [空 inode 槽位]
Block 2052: [root目录项: "."(ino=1), ".."(ino=1), "test.txt"(ino=2)]
Block 2053: [文件数据: "Hello MiniFS!\nThis is read-only FUSE demo.\n"]
Block 2054~16383: [空数据块]
```

> **注意**：根目录 nlink=3，而非通常的 2，因为 test.txt 不影响 nlink，但 dir1 的 `..` 会 +1。此处 nlink=3 是因为格式化时只创建了 root 和 test.txt，root 的 nlink 应为 2（`.` 和 `..`），代码中设置为 3 是一个潜在 bug。

## Q: 在 /dir1 目录下创建新文件 1.log（含内容），需要几次磁盘读写？

### 前置：路径解析（/ → dir1）

| 步骤 | 操作 | 磁盘读 | 磁盘写 | 说明 |
|------|------|--------|--------|------|
| 1 | 读取根 inode | 1 | 0 | 找到根目录数据块号 `direct[0]` |
| 2 | 读取根数据块 | 1 | 0 | 遍历目录项，找到 `dir1` 的 inode 号 |
| 3 | 读取 dir1 inode | 1 | 0 | 找到 dir1 数据块号 `direct[0]` |
| 4 | 读取 dir1 数据块 | 1 | 0 | 读取目录项，准备追加 `1.log` 条目 |

### 文件创建

| 步骤 | 操作 | 磁盘读 | 磁盘写 | 说明 |
|------|------|--------|--------|------|
| 5 | 分配 1.log 的 inode | 1 | 1 | `alloc_inode`：读 inode 位图 → 找空闲 bit → 写回位图 |
| 6 | 写入 1.log 的 inode | 0 | 1 | mode=0100644, nlink=1, size=内容长度 |
| 7 | 分配 1.log 的数据块 | 1 | 1 | `alloc_data_blk`：读数据位图 → 找空闲 bit → 写回位图 |
| 8 | 写入 1.log 的数据块 | 0 | 1 | 写入文件内容 |

### 更新父目录

| 步骤 | 操作 | 磁盘读 | 磁盘写 | 说明 |
|------|------|--------|--------|------|
| 9 | 写入 dir1 数据块 | 0 | 1 | 追加 `1.log` 目录项后写回 |
| 10 | 写入 dir1 inode | 0 | 1 | 更新 size、mtime、ctime |

### 汇总

| 阶段 | 磁盘读 | 磁盘写 |
|------|--------|--------|
| 路径解析 | 4 | 0 |
| 文件创建 | 2 | 4 |
| 更新父目录 | 0 | 2 |
| **合计** | **6** | **6** |
| **总 I/O** | **12** | |

### dir1 inode 需要更新的字段

- `size` += `sizeof(struct dirent)`（130 字节）
- `mtime` = 当前时间（目录内容变更）
- `ctime` = 当前时间（元数据变更）
- **注意**：`nlink` **不变**，因为普通文件不会增加父目录的硬链接数（与创建子目录不同）

### 对比：创建空文件（无内容）

如果只是 `touch` 创建空文件（size=0），无需分配数据块和写入内容，可省略步骤 7 和 8：

| 类型 | 次数 |
|------|------|
| 磁盘读 | 5 |
| 磁盘写 | 4 |
| **总 I/O** | **9** |

## Q: 在根目录下创建一个新目录 dir1，需要涉及几次磁盘读写？

### 操作步骤与 I/O 明细

| 步骤 | 操作 | 磁盘读 | 磁盘写 | 说明 |
|------|------|--------|--------|------|
| 1 | 读取根 inode | 1 | 0 | `read_inode(fd, root_ino)`，获取根目录数据块号 `direct[0]` |
| 2 | 读取根数据块 | 1 | 0 | 读取根目录的目录项，准备追加 `dir1` 条目 |
| 3 | 分配 dir1 的 inode | 1 | 1 | `alloc_inode`：读 inode 位图 (block 2) → 找空闲 bit → 写回位图 |
| 4 | 写入 dir1 的 inode | 0 | 1 | `write_inode(fd, dir1_ino, &dir1_in)`，mode=040755, nlink=2 |
| 5 | 分配 dir1 的数据块 | 1 | 1 | `alloc_data_blk`：读数据位图 (block 3) → 找空闲 bit → 写回位图 |
| 6 | 写入 dir1 的数据块 | 0 | 1 | 写入 `.`（指向 dir1 自身）和 `..`（指向根目录）两个目录项 |
| 7 | 写入根数据块 | 0 | 1 | 追加 `dir1` 目录项后写回 |
| 8 | 写入根 inode | 0 | 1 | 更新 size（+130B）、nlink（+1，因为 dir1 的 `..` 指向根）、mtime、ctime |

### 汇总

| 类型 | 次数 |
|------|------|
| 磁盘读 | **4** |
| 磁盘写 | **6** |
| **总 I/O** | **10** |

### 根 inode 需要更新的字段

- `size` += `sizeof(struct dirent)`（130 字节）
- `nlink` += 1（dir1 的 `..` 指向根目录，根的硬链接数 +1）
- `mtime` = 当前时间（目录内容变更）
- `ctime` = 当前时间（元数据变更）

> 对比创建普通文件，区别在于：新目录需要额外分配数据块并写入 `.` 和 `..`，且父目录的 `nlink` 需要 +1；创建普通文件不会增加父目录的 `nlink`。

## Q: 单个目录下最多创建多少个文件和目录？

### 每个数据块能存多少个目录项

`struct dirent` 的布局（packed）：

| 字段 | 大小 |
|------|------|
| `ino` (uint32_t) | 4 字节 |
| `name` (char[124]) | 124 字节 |
| `rec_len` (uint16_t) | 2 字节 |
| **合计** | **130 字节** |

每个数据块 4096 字节：

```
floor(4096 / 130) = 31 个目录项/块
```

剩余 4096 - 31×130 = 66 字节，不足以再放一个目录项，浪费在块末尾。

### 单个目录最多寻址多少个数据块

inode 结构中只有直接块指针：

```c
uint32_t direct[12]; // 直接块指针
```

**没有间接块指针**，因此一个目录最多使用 **12 个数据块**。

### 结论

| 项目 | 数量 |
|------|------|
| 每块目录项 | 31 |
| 最大数据块 | 12 |
| 总目录项 | 12 × 31 = **372** |
| 固定占用（`.` 和 `..`） | 2 |
| **用户可创建** | **370** |

瓶颈在 **12 个直接块指针 × 31 条/块 = 370 个文件/子目录**。

> 若需支持更多，需增加间接块指针（single indirect → double indirect → triple indirect），这也是 ext2/3 等真实文件系统的做法。
